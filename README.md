# AIエージェントスキル仕様書：カード型長期記憶管理（LLM Wiki）

本仕様書は、AIエージェントがユーザーとの対話から得た知識を「カード」という単位で構造化し、長期記憶として保持・管理するための拡張機能（スキル）の仕様を定義するものです。

## 1. 概要 (Overview)

通常のテキストベースのWikiや単純なチャット履歴とは異なり、記憶を「カード」という構造化された最小単位にパッキングします。各カードにカテゴリやタグ、そしてカード同士の相互リンク（Relations）を持たせることで、人間にとって直感的で管理しやすく、かつAI（LLM）にとっても検索・関連付けが容易な長期記憶ネットワークを構築します。

### 核心的なメリット
- **コンテキストの節約**: 過去の全履歴をLLMに送るのではなく、必要なカードだけをオンデマンドで引き出すため、トークン消費を抑えられます。
- **知識の構造化**: 雑多な会話から重要なファクトやユーザーの好みを抽出し、整理されたデータベースに昇華させます。
- **直感的なユーザー体験**: ユーザー自身がカードの一覧や繋がりをUI上で一目で確認・メンテナンスできます。

## 2. 核心的な振る舞いと運用フロー (Core Workflow)

### 2.1. 記憶の蓄積・更新（提案型ハイブリッド）

記憶の新規登録や既存カードのアップデートは、AIによる「自発的提案」とユーザーからの「明示的コマンド」のハイブリッドで実現します。

- **AIの自発的提案**: 会話の中で「重要な情報（ユーザーの好み、決定事項、プロジェクトのアイデアなど）」を検知した場合、AIは自動でカードのドラフト（タイトル、本文、タグ等）を裏で生成し、会話内でユーザーに提案します。
  - *例*: 「『Next.jsプロジェクトの技術スタック』について重要な決定があったため、カードを作成しましょうか？」
- **ユーザーによる明示的コマンド**: ユーザーが能動的に「今のアイデアを『〇〇』というカードで保存しておいて」と指示した場合も、同様にドラフトを作成して保存処理を実行します。
- **重複・競合時の挙動（マージ提案）**: 新しく得た情報がすでに存在しているカードと類似している、または既存情報のアップデートであるとAIが判断した場合、新規作成は行わず、既存カードへの自動マージ（上書き）案をユーザーに提示します。
  - *例*: 「既存のカード『旅行の計画』に、新しく決まった『ホテルのチェックイン時間』を追記してアップデートしますか？」

### 2.2. 定期的な記憶の整理（メンテナンスモード）

カードの肥大化や陳腐化を防ぐため、AI主導で定期的なクリーンアップを行います。

- **起動トリガー**: 定期的（例：一定会話数ごと、またはユーザーからの明示的な「記憶を整理して」という指示）にバックグラウンドで起動します。
- **挙動**: AIがすべてのカードの依存関係や重複をスキャンし、「重複しているカード」「不要になった古いカード」「1つに統合した方が体系的になるカード」を特定します。
- **ユーザーへの確認**: 実際に削除・統合を行う前に、AIは「どのカードが消去され、どのカードにマージされるか」のBefore/Afterを明確に提示し、ユーザーの同意（承認）を得てから実行します。

## 3. データ構造 (Data Model)

各記憶カードは、以下の項目を持つ構造化データ（JSON形式）として永続化されます。

```json
{
  "id": "string (UUID v4)",
  "title": "string (一目で内容がわかる短い要約。例: '私のコーヒーの好み')",
  "content": "string (Markdown形式の本文。表やリスト等も含められる)",
  "category": "string (大分類。例: '技術', '個人', '仕事')",
  "tags": [
    "string (小分類・キーワード。例: '浅煎り', 'エチオピア')"
  ],
  "created_at": "string (ISO 8601形式: YYYY-MM-DDTHH:mm:ssZ)",
  "updated_at": "string (最終更新日時。鮮度管理やマージの基準に使用)",
  "relations": [
    "string (関連する他のカードのID配列。カード間のネットワークを形成)"
  ]
}
```

## 4. AI向けインターフェース / ツール定義 (Function Calling API)

AIエージェントが会話中にこのスキルを操作するために呼び出す関数（Tools）の定義です。

### 4.1. search_cards (オンデマンド検索)

AIが必要な時に、自発的に過去のカードを検索して記憶を思い出すためのツール。

#### 引数（パラメータ）:
- `query` (string, optional): 検索クエリ（セマンティック検索やキーワード検索に用いる文字列）
- `category` (string, optional): カテゴリによる厳密フィルタリング
- `tags` (array of strings, optional): 指定したタグをすべて、または一部含むカードのフィルタリング

#### 返り値:
- マッチしたカードオブジェクトの配列。AIはこの内容をコンテキスト（システムプロンプトや追加情報）に挿入して回答を生成します。

### 4.2. propose_card_action (作成・変更の提案)

AIが新規作成、または既存カードへの上書き（マージ）のドラフトを生成し、システム（およびUI）に送信するツール。

#### 引数（パラメータ）:
- `action_type` (string): `"create"`（新規作成） または `"update"`（既存更新/マージ）
- `target_card_id` (string, optional): `action_type` が `"update"` の場合のみ必須。更新対象のカードID
- `draft_data` (object): 作成または更新するカードデータのドラフト（`title`, `content`, `category`, `tags`, `relations` を含む）

#### 返り値:
- 提案が正常にUIへ送信されたかどうかのステータス。

### 4.3. propose_memory_cleanup (記憶整理の提案)

重複したカードや古いカードの統合・削除案をまとめて提示するメンテナンス用ツール。

#### 引数（パラメータ）:
- `delete_targets` (array of objects): 削除対象のカードリスト（各要素は `{ id: string, title: string, reason: string }`）
- `merge_targets` (array of objects): 統合対象のリスト。以下のような構造：

```json
[
  {
    "source_ids": ["string (統合されて消えるカードのID)"],
    "destination_id": "string (統合先として残るカードのID。新規作成の場合はnull)",
    "proposed_card": {
      "title": "string (統合後の新しいタイトル)",
      "content": "string (マージされて整理された新しい本文案)",
      "category": "string",
      "tags": ["string"],
      "relations": ["string"]
    },
    "reason": "string (なぜ統合を推奨するのかの理由)"
  }
]
```

#### 返り値:
- クリーンアップ案がユーザーに送信された成否。

## 5. UI/UX・表示制御要件 (UI/UX & Rendering Specs)

### 5.1. 外部アクセス性とURLの自動リンク化 (Autolinking & Standard Links)
- **標準リンクのサポート**: `[表示テキスト](URL)` という標準的なマークダウンリンク記法を検出し、自動的に `target="_blank"` 属性を持ったクリック可能な `<a>` タグへパース・レンダリングします。
- **生URLの自動リンク（Autolink）**: 文章内の生のURL（`https://...` または `http://...`）を自動的に検出し、クリック可能なリンクへ変換します。
- **パース時の堅牢性**: インラインコード（` `code` `）やHTML属性内などの境界文字（バッククォート、ダブルクォーテーション等）をURLの一部として誤認識しないよう、正規表現や字句解析の境界を厳密に処理します。

### 5.2. カードの動的ソートとUI並び替え (Sorting & Dynamic Reordering)
- **ソート基準の選択**: ユーザーが情報を迅速に見つけられるよう、カードの表示順を動的に切り替えるインターフェースを提供します。
  - **新しい順 (newest)**: 更新日時 (`updated_at`) が新しい順。デフォルトの初期表示順。
  - **古い順 (oldest)**: 更新日時 (`updated_at`) が古い順。
  - **タイトル順 (title)**: タイトル (`title`) の辞書順（日本語の五十音順およびアルファベット順）。
- **トランジション効果**: 表示順の変更や詳細パネルの開閉時に、カードの配置変更やグリッドの伸縮がガタつくのを防ぐため、ブラウザの View Transitions API 等を用いて、カードが滑らかに位置移動するアニメーションを適用します。

### 5.3. データ更新の即時反映（キャッシュ無効化）
- **動的フェッチの強制**: データベースファイル（`db.json`等）への変更がブラウザのキャッシュによって古い情報のまま表示されるのを防ぐため、データ取得API（`/api/cards`）は静的キャッシュを強制的に無効化（`force-dynamic`等）し、常にオンデマンドで最新のデータファイルを読み込みます。

## 6. 今後の拡張アイデア (Roadmap)

- **セマンティック・ベクトル検索の導入**: 単なるキーワード一致ではなく、文脈や意味の類似性で関連カードを自動検出する。
- **外部ツール連携**: 作成されたカード（Markdown）を自動的にObsidianやNotionなどの既存の外部ノートツールへ同期する。
